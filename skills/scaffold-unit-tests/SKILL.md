# Scaffold Unit Tests

Generates failing Jest test stubs for a source module. Tests are written **before** implementation — they must fail first. This is the floor: the tests define what the implementation must prove.

## Step 1: Read Foundation

1. Read `plans/permanent/codebase-index.md` → `test_root` and `test_pattern`
2. Read `plans/permanent/glossary.md` → domain terms for naming
3. Identify the target source file (specified by user, or from the current feature plan)

## Step 2: Determine the Test File Path

Mirror the source path using the test root from the index:

| Source | Test root | Test file |
|--------|-----------|-----------|
| `src/domain/order.js` | `spec/` | `spec/domain/order.spec.js` |
| `src/application/checkout.js` | `__tests__/` | `__tests__/application/checkout.test.js` |

Create intermediate directories if they don't exist.

If a test file already exists, read it first and only add stubs for functions that don't have a test yet — never overwrite existing tests.

## Step 3: Analyze the Source Interface

Read the source file. Extract:
- All exported functions and their parameter signatures
- Any exported classes and their public methods
- Domain invariants visible in the code (guards, validations, throws)
- Return types or shapes implied by the code

## Step 4: Generate Test Stubs

For each exported function or class method, generate a `describe` + `it` block:

```javascript
describe('<FunctionName / ClassName>', () => {

  it('should <observable outcome in domain language> when <condition>', () => {
    // Arrange

    // Act

    // Assert
    throw new Error('Not implemented');
  });

  it('should reject <invalid input or wrong state> when <condition>', () => {
    // Arrange

    // Act

    // Assert
    throw new Error('Not implemented');
  });

});
```

**For every function, write at minimum:**
1. The happy path test
2. At least one failure/rejection test
3. Any boundary tests implied by visible guards in the source

## Step 5: Naming Rules (Enforced)

Test names MUST describe behavior in domain language:

| ✓ Good | ✗ Bad |
|--------|-------|
| `should reject an order when the cart is empty` | `should return false` |
| `should confirm an order when all items are in stock` | `should call placeOrder` |
| `should calculate the correct total when a discount applies` | `test_calculateTotal` |
| `should throw when the user account is suspended` | `should throw an error` |

Use terms from `plans/permanent/glossary.md`. If a concept doesn't have a glossary term, flag it.

## Step 6: Write the File

Output the complete test file with:
```javascript
'use strict';

/**
 * Unit tests for: <source file path>
 * Generated by scaffold-unit-tests skill — implement each test body.
 * All tests should fail until the implementation is complete.
 */

// TODO: Replace with actual import
const { /* exports */ } = require('../<relative-path-to-source>');

describe('<ModuleName>', () => {

  // --- generated stubs ---

});
```

## Step 7: Verify All Tests Fail

Run the test file to confirm all stubs fail with "Not implemented" (not a runtime error):
```bash
npx jest <test-file-path> --no-coverage 2>&1 | tail -20
```

If any test passes without implementation, something is wrong — report it.

## Step 8: Report

```
Scaffolded: spec/domain/order.spec.js
  • 3 describe blocks
  • 8 test stubs (all failing ✓)
  • 2 glossary terms used: Order, Cart
  • ⚠ 1 term not in glossary: 'stockReservation' — consider adding it

Next steps:
  1. Run acceptance pipeline to see failing acceptance tests
  2. Implement the module to make tests pass
  3. Run run-crap-analysis once tests are green
```

Update `plans/<feature-name>.md`: check off `scaffold-unit-tests` in the Pipeline Status block.
